Low Budget Open Source Home Server // home infrastructure
Dell Wyse 5070 thin client re-purposed to a home server, with 4TB of attached HDD storage.

Running services:

  • Immich — photo management
  • Tailscale — remote access
  • Pi-hole — network ad blocking
  • Home Assistant — smart home
  • Samba + Filebrowser — file server
  • Rotating offsite backup system
▶ [01] Working Method

All configuration and maintenance is tracked through a three-layer documentation system, designed to keep complex infrastructure legible across months of intermittent work.

▶ Central config index

A single master file (Wyse5070Server-Config.md) that acts as the table of contents for the entire system. Kept intentionally short — component status, service URLs, quick-reference commands, and links to each subsystem file. The entry point for any session; never grows beyond its purpose.

▶ Per-subsystem config files

One dedicated file per service — Tailscale, Pi-hole, Home Assistant, file server, backup system, and hardware. Each contains the full technical detail for that subsystem: config paths, relevant options explained, and any non-obvious decisions. Alongside these, a shared known-issues reference lists recurring symptoms with quick fixes and pointers back to the session log where each was resolved. Only the file relevant to the current session needs to be loaded.

▶ Session logs

Every working session gets a dated entry: what was done, what was tried, root cause if something broke, and what was learned. This is the layer that answers the question config files can't — why does this look like this? It also surfaces patterns: recurring failures or issues that seem unrelated until the log shows they're not.

▶ [02] Hardware & Drive Management

€57 of secondhand hardware running 24/7 at ~10W. Every component chosen or upgraded for a specific reason.

▶ Hardware
  • Dell Wyse 5070 thin client (~€30 used). Silent, built for 24/7. Intel Celeron J4105, 8GB DDR4, Gigabit Ethernet. Upgraded from eMMC to 128GB M.2 SATA SSD (€13) for speed and storage headroom.
  • 4TB Seagate Barracuda 2.5" — data storage, 1.8TB in use. Connected over USB 3.0. Two rotating 5TB USB drives for offsite backup (one home, one travels).
  • StarTech USB 3.0 SATA adapter (0x14b0:0x0206) — replaced the original Seagate enclosure specifically to get SMART passthrough support via SAT protocol.
▶ A 2.5" External USB Hard Drive Has No Business Being on a Server

For a home server with hours of idle between accesses, a laptop drive is actually a reasonable fit — the usage pattern is closer to a laptop than a rack server — sessions of a few hours with sometimes days of idle in between — the cost is a fraction, and the drive was already available. The catch: it needs to be actively spun down between uses to protect it, and keeping it down turned out to be unexpectedly complex. USB also adds a bridge chip that quietly breaks several things.

▶ bug SMART data inaccessible through USB bridge
symptomStandard smartctl /dev/sdb returns no data — USB bridges don't forward ATA commands by default.
root causeThe bridge chip translates SCSI commands but drops most ATA passthrough. SMART requires direct ATA access via SAT (SCSI ATA Translation) protocol.
fixReplaced original Seagate enclosure with a StarTech USB-SATA adapter (0x14b0:0x0206) which supports SAT passthrough. Command: smartctl -a -d sat /dev/sdb
side effectAfter sustained I/O, the bridge leaves hdparm -C permanently unreliable. Switched spin-state detection to smartctl -n standby (SAT passthrough, never wakes drive). Exit 0 = spinning, exit 2 = standby.
▶ bug HDD spinning ~25× per day with no user activity
symptomStart_Stop_Count +25×/day. Drive spinning down every 30–60 seconds, then waking on any activity.
root causeThe original Seagate enclosure's firmware disabled the drive's aggressive factory standby timer on init. The StarTech adapter is a dumb pass-through — it doesn't configure anything, so the drive reverted to factory defaults: spin down after ~30 seconds of idle.
fixhdparm -S 0 -B 254 to disable the standby timer. Made persistent via udev rule — settings re-applied on every USB reconnect.
▶ bug Drive wakes within seconds of every spin-down attempt
contextAfter disabling the standby timer, tried to re-enable controlled spin-down with a 1-hour idle timer. Drive woke within 2–3 seconds of every attempt.
methodfatrace /mnt/storage soak monitoring while drive was supposed to be idle.
hypothesissystemd-journald: sync() every 5 min could trigger ext4 journal commits on all filesystems. Applied Storage=volatile precautionarily — drive still woke.
culpritWebmin: scheduled_collect_system_info runs df on all filesystems every 5 minutes, even with no active web session.
fixStorage=volatile in journald.conf + systemctl disable --now webmin
verified5.5-hour zero-access soak test. Start_Stop_Count delta: +1 (the SMART check itself).
▶ bug HDD wakes on first access after >24h idle
symptomAfter the journald + Webmin fix, drive still woke on first SSHFS mount after a long absence.
constraintOnly reproducible after >24h away — impossible to catch in normal daily-use testing.
root causeThe relatime mount option (included in defaults) updates atime when the stored value is older than 24h. Creates a dirty inode that ext4's 5-second journal commit interval flushes to disk — waking the HDD.
fixAdded noatime to /etc/fstab mount options. Applied live with mount -o remount,noatime — no reboot needed.
▶ note APM 127 vs 128 — one value that changes everything

APM (Advanced Power Management) values 1–127 each carry their own internal idle timer that overrides the -S standby setting entirely. APM 127 caused the drive to spin down within 2 minutes regardless of what hdparm -S was set to. APM 128 disables APM's internal timer while still allowing the -S standby timer to work. Production setting: hdparm -B 128 -S 242 (APM 128, 1-hour standby). Persistent via udev rule at /etc/udev/rules.d/69-hdparm-sdb.rules.

▶ Drive Health Monitoring

Consumer hardware — health monitoring is critical. Two systems run continuously: a weekly SMART check for long-term trends, and a real-time spin state monitor. Both push phone notifications.

▶ note Weekly SMART check — Sunday 3 AM
  • Runs every Sunday 3:00 AM via systemd timer. Tracks key attributes with week-on-week deltas: Start/Stop Count, Spin_Up_Time (arbitrary units — a rising trend is an early motor wear indicator), Power-On Hours, Head Flying Hours, temperature, reallocated/pending sectors.
  • Results pushed to Home Assistant via webhook → dashboard showing last 30 checks.
  • All weekly maintenance aligned to the same window: SMART 3:00, e2scrub 3:10, SSD TRIM 3:15. Drive wakes once, then spins back down after an hour.
  • Script re-applies hdparm -B 128 -S 242 after each run — USB bridge leaves power settings in an inconsistent state after SMART access, which blinds the spin monitor until settings are re-applied.
▶ note Real-time spin state monitor
  • Persistent systemd service polling every 30 seconds. On state change (spinning ↔ standby) → HA webhook → phone notification.
  • Uses smartctl -d sat -n standby rather than hdparm -C — after sustained I/O the USB bridge makes hdparm -C permanently unreliable. SAT passthrough stays consistent and never wakes the drive.
  • Spin history (last 50 entries) surfaced in the HA dashboard alongside SMART data.
▶ [03] Tailscale — Remote Access

Private WireGuard network that makes the server accessible from anywhere over the internet as if on the local network — no VPN server to manage.

▶ Security

Access is controlled per device via ACLs — new devices get zero access by default. The laptop is the only device with full access to all services. Every other device is restricted to exactly the services it needs.

▶ Performance

Tailscale was falling back to relay servers (~5 MB/s) — the router's NAT configuration was blocking direct hole-punching. Adding a port forwarding rule gave Tailscale a stable inbound endpoint. Result: a steady 52 MB/s — the bottleneck is the server's CPU doing WireGuard encryption, not the network.

▶ [04] File Server

The original reason for the server — 4TB of storage accessible from any device, anywhere.

▶ Access

Two interfaces cover all use cases. The laptop mounts via SSHFS on demand using a desktop toggle; the iPhone connects through Samba via the iOS Files app. Filebrowser provides a web UI for remote access and file sharing via link. Local transfers run at ~100 MB/s over the home network; remote via Tailscale at ~52 MB/s.

▶ bug Three auto-mount approaches failed — ended up with a manual toggle
goalLaptop mounts server storage automatically on connect, unmounts cleanly on suspend/shutdown.
attempt 1fstab CIFS + x-systemd.automount — GVFS retry loop when off-site, Nemo sluggish.
attempt 2NetworkManager dispatcher CIFS — D-state kernel freeze on suspend. CIFS uses kernel-space blocking I/O; can't be interrupted during suspend.
attempt 3NM dispatcher SSHFS (FUSE) — Nemo/Cinnamon sidebar polls the FUSE mount via GIO continuously. Caused freezes and one Cinnamon crash.
root causeAny permanent FUSE mount in Nemo's sidebar triggers constant GIO/GVFS polling — unavoidable with auto-mount.
fixManual desktop toggle — the right fit for on-demand use. No persistent mount, no background polling, no GIO watching a remote filesystem. Sleep hook and shutdown service handle clean unmount automatically.
▶ [05] Pi-hole — Network Ad Blocking

DNS-based ad blocker running on the server, blocking ads across all Tailscale-connected devices — including inside apps on iOS, on mobile data, and on foreign WiFi, not just at home.

▶ How it works

Pi-hole intercepts DNS queries and drops requests for known ad domains — 86,832 blocked by default (StevenBlack list). Devices opt in by enabling Tailscale DNS in the Tailscale app, which routes all their DNS through the server's Pi-hole. Because it runs over Tailscale, it works anywhere the device has a connection.

All devices get ad-blocking via DNS (port 53 open to everyone in the ACL), but only the laptop and one phone can access the Pi-hole dashboard to change settings. The server itself uses a direct upstream resolver.

▶ [06] Home Assistant — Smart Home

Self-hosted home automation hub running in Docker on the server — controls lights, sensors, and outlets from anywhere in the world via Tailscale, and doubles as the monitoring dashboard for the server itself.

▶ Integrations

IKEA DIRIGERA hub connected via Matter protocol — lights and outlets controllable from HA. Apple HomeKit bridge runs in parallel, so Siri and the Home app work simultaneously without any conflict. Tailscale IPs are configured as trusted networks — HA skips authentication for connections from those IPs, so remote access via Tailscale requires no login.

▶ Automations

Lights turn on automatically when someone arrives home after dark — triggered by phone location via the HA mobile app. The automation has a prioritised fallback chain: if the primary light is unreachable (switch accidentally turned off, device unavailable), it tries the next one down the list rather than failing silently. Arriving home always resets the primary light to full brightness regardless of its previous state.

▶ Server monitoring dashboard

HA serves as the frontend for all server drive monitoring — SMART check history (last 30 runs with per-attribute deltas), real-time spin state, spin history, and manual trigger buttons for SMART checks and forced spin-down. Data is pushed from the server via webhooks and read by command_line sensors polling shell scripts every 60 seconds. All of it visible on mobile from anywhere via Tailscale.

▶ [07] Immich — Photo Management

Self-hosted Google Photos alternative — automatic phone backup, AI-powered face and scene recognition, full photo library accessible from anywhere via Tailscale. No subscriptions, no third-party cloud.

▶ Setup

Four Docker containers (server, machine learning, database, Redis). The machine learning container handles face clustering and scene detection locally — no data leaves the server. Photos stored on the 4TB HDD, database on the SSD — faster queries and write-heavy ML indexing stays off the HDD.

▶ [08] Backup System

Restic + Backrest covering both server and laptop, with two rotating 5TB drives — one onsite, one offsite, swapped on every trip.

▶ Strategy & Architecture

Restic is an incremental, deduplicated, encrypted backup tool. Backrest is a web GUI on top of it. The backup system was built and verified before any data was migrated — three copies: live data on the server, one drive onsite, one offsite. Each machine backs up independently using its own Backrest instance, so either drive works for either machine when plugged in.

▶ Drive Verification

Every drive is burnt in before being trusted with backup data: full dd write across the entire surface, read-back verification, then restic check --read-data (full SHA-256 data verification). The process exists because of this:

▶ bug WD Elements 5TB — 18,971 bad blocks discovered after being trusted with backups
symptomDrive appeared healthy — SMART showed only 8 bad sectors during normal read-only use.
root causeWriting new data corrupted sectors that were previously readable — invisible to read-only SMART tests.
discoveredDestructive badblocks write test revealed 18,971 bad blocks across two physical zones. Drive condemned and retired.
lessonRead-only SMART is not sufficient. A full write + read-back test (dd fill → verify → restic check) is required before trusting any drive with backup data.
▶ Issues & fixes
▶ bug Every server backup was a full 1.7TB re-read — 28h+ per run
symptomEach backup re-read the full 1.7TB of source data regardless of what had changed — 28h+ per run.
root causeServer data was read via SSHFS from the laptop. SSHFS inode numbers are unstable across remounts — Restic uses inodes to detect unchanged files, so every file looked new every time.
fixMoved server backups to run on the server itself — local I/O, no network. Restic now sees stable inodes. Speed: 17 MB/s → 50–100 MB/s. Incrementals now only process actual changes.
